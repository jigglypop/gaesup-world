# Editor Camera and Control Cleanup Plan

에디터 1차 정리의 목표는 카메라와 캐릭터 컨트롤이 서로 덮어쓰는 문제를 줄이고, 에디터 진입/종료 시 월드 상태가 예측 가능하게 복원되도록 만드는 것입니다.

이 문서는 대규모 구조 변경 계획이 아니라, 현재 코드 위에서 안전하게 시작할 수 있는 1차 정리 범위를 정의합니다.

## 배경

현재 `Editor`는 마운트 시 `useGaesupStore`의 `mode`와 `cameraOption`을 에디터용 값으로 변경하고, 언마운트 시 이전 값을 복원합니다.

관련 위치:

- `src/core/editor/components/Editor/index.tsx`
- `src/core/editor/components/EditorLayout/index.tsx`
- `src/core/stores/gaesupStore`
- `src/core/stores/slices/mode`
- `src/core/camera`
- `src/core/interactions`
- `examples/pages/World.tsx`

이 구조는 단순하지만, 에디터와 플레이 모드가 같은 전역 store를 공유하기 때문에 다음 문제가 생기기 쉽습니다.

- 에디터 진입 시 플레이어 컨트롤 모드가 예기치 않게 바뀜
- 에디터 종료 후 카메라 옵션이 이전 상태로 완전히 돌아오지 않음
- 월드 페이지, 어드민 페이지, 편집 페이지마다 `showEditor`/`showHud` 조합이 달라 상태 복원 타이밍이 불명확함
- 카메라 변경 책임이 `Editor`, 패널, 월드 페이지에 분산될 수 있음

## 원칙

- 기존 store와 공개 API를 먼저 보강합니다.
- 카메라, 캐릭터 컨트롤, 에디터 UI 책임을 섞지 않습니다.
- 에디터 진입/종료는 한 곳에서 관리합니다.
- 불필요한 파일명 변경이나 함수명 변경은 하지 않습니다.
- 큰 분리보다 작은 안정화 작업을 먼저 합니다.

## 1차 목표

### 1. 에디터 상태 스냅샷 명확화

`Editor`가 변경하는 전역 상태를 명확히 기록합니다.

우선 대상:

- `mode`
- `cameraOption`

검토 대상:

- 입력 잠금 여부
- 포인터/클릭 배치 상태
- 현재 선택된 빌딩 편집 모드
- HUD와 상호작용 prompt 표시 여부

완료 기준:

- `Editor`가 직접 변경하는 상태 목록이 코드에서 한눈에 보임
- 언마운트 시 복원 대상이 누락되지 않음
- 복원 로직이 early return 중심으로 단순하게 유지됨

### 2. 에디터 카메라 프리셋 분리

현재 `EDITOR_CAMERA_OPTION`은 `Editor/index.tsx` 내부 상수입니다. 1차에서는 파일 이동을 필수로 하지 않되, 다음 기준으로 정리합니다.

- 에디터 카메라 값은 플레이 카메라 기본값과 분리
- top-down 편집에 필요한 값만 포함
- 기존 카메라 옵션 전체를 새 값으로 덮어쓸지, 필요한 필드만 merge할지 정책 확정

권장 정책:

- 에디터 진입 시에는 필요한 필드만 override합니다.
- 종료 시에는 진입 전 스냅샷을 그대로 복원합니다.

완료 기준:

- 에디터 카메라 프리셋의 의도가 주석 없이도 드러남
- 플레이 모드 카메라 설정과 섞이지 않음

### 3. 캐릭터 컨트롤 잠금 정책 결정

에디터가 켜졌을 때 캐릭터가 움직여야 하는지 정책을 고정합니다.

권장 정책:

- 월드 편집 중에는 캐릭터 이동 입력을 막습니다.
- 카메라 이동, 배치, 선택 입력은 에디터가 담당합니다.
- 플레이 테스트 모드가 필요하면 별도 모드로 분리합니다.

완료 기준:

- `showEditor` 상태에서 캐릭터 이동과 에디터 입력이 충돌하지 않음
- `/world`, `/edit`, `/admin`의 동작 차이가 코드에서 명확함
- 임시 로그나 더미 상태 없이 코드 분석으로 흐름을 확인할 수 있음

### 4. `examples/pages/World.tsx`의 모드 분기 정리

`WorldPage`는 데모의 중심이므로, 에디터 모드와 플레이 모드의 차이를 여기서 명확히 확인합니다.

확인할 조합:

- `/`: showcase
- `/world`: HUD 중심 플레이
- `/edit`: 에디터 중심 편집
- `/admin/*`: 어드민으로 감싼 에디터

완료 기준:

- `showEditor`가 true일 때 클릭 이동, 지면 클릭, 배치 입력이 서로 충돌하지 않음
- `showHud`와 `showEditor`가 독립적으로 동작함
- 페이지 언마운트 후 전역 상태가 다음 페이지에 새지 않음

## 작업 순서

1. `Editor/index.tsx`에서 현재 변경하는 store 상태를 정리합니다.
2. 에디터 진입 스냅샷 타입을 명확히 합니다.
3. 에디터 종료 복원 로직을 작고 확실하게 유지합니다.
4. `WorldPage`에서 `showEditor`일 때 캐릭터/클릭 컨트롤이 어떻게 꺼지는지 확인합니다.
5. 필요한 경우 입력 잠금 또는 에디터 활성 상태를 기존 store에 보강합니다.
6. `/edit`, `/world`, `/admin` 기준으로 동작을 검증합니다.

## 하지 않을 일

- 빌딩 패널 전체 리팩터링
- NPC store 분리
- 카메라 시스템 전체 재작성
- 새 기술 스택 도입
- 임시 더미 페이지나 임시 파일 생성
- 함수명만 바꾸는 정리

## 검증 기준

코드 수정 후 아래 순서로 확인합니다.

1. `npm run lint`
2. `npm run build:types`
3. 필요 시 관련 테스트 실행
4. `npm run dev`로 `/edit`, `/world`, `/admin` 수동 확인

수동 확인 항목:

- 에디터 진입 시 카메라가 top-down 편집 시점으로 전환됨
- 에디터 종료 후 이전 카메라/컨트롤 상태가 복원됨
- 에디터 모드에서 캐릭터 이동 입력이 편집 입력과 충돌하지 않음
- 월드 플레이 모드에서 기존 캐릭터 컨트롤이 유지됨

## 후속 단계

1차 정리가 끝난 뒤에는 다음 순서로 확장합니다.

- `BuildingPanel`의 카메라/선택/배치 책임 분리
- NPC 편집 UI를 건축 패널에서 분리할지 검토
- 에디터 전용 store와 실제 패널 상태 연결 여부 결정
- 저장/복원 흐름에서 에디터 상태가 world save에 섞이지 않도록 정책화
